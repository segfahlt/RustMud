# LLM Integration Design

Status: **Exploratory** — no implementation yet

This document captures thinking about how LLM/AI APIs could make RustMud feel meaningfully different from a traditional MUD. The goal isn't to replace MUD mechanics with AI — it's to use AI where variability and language generation add value that static text cannot.

---

## Guiding Constraints

- **Latency**: MUD players expect near-instant responses for movement and commands. LLM calls (0.5–3s) are acceptable only where the player expects a pause: NPC dialogue, lore queries, quest generation.
- **Cost**: high-frequency calls (room descriptions on every move) need either caching, local models, or selective invocation.
- **Grounding**: the LLM must operate within defined world facts. It generates language, not truth. All structural data (item existence, room exits, mob stats) comes from the game engine, never from the model.
- **Failure safety**: if an LLM call fails or times out, the game falls back to static text gracefully.

---

## Use Cases

### 1. Dynamic NPC Dialogue

**What**: Players can `talk to <mob>` and have a freeform conversation. The NPC responds in character, drawing on a persona and lore document defined in their mob file.

**Prompt structure**:
```
System: You are {mob.name}, a {mob.description}. 
World context: {lore_snippet}
Personality: {mob.personality}
Rules: Stay in character. Do not break the fourth wall. 
       Do not invent quests or items that don't exist in the game data.
       Keep responses under 3 paragraphs.

Player says: "{player_input}"
```

**Implementation notes**:
- Mob file gains `dialogue: { persona: "...", lore_ref: "..." }` field
- Conversation history per session (not persisted across login)
- Hard cutoff: 6 turns max, then NPC says they have work to do
- Static fallback: if no `dialogue` field, `speak` gives a canned response

### 2. Procedural Area and Room Generation

**What**: Areas (outdoor spaces) and Rooms (buildings) are generated on demand as players explore. Each generation call produces a description, a set of exit hints, and a fixture list appropriate to the location's zone state and evolution stage.

**When to call**:
- Player steps into an unvisited Area or Room for the first time (primary trigger)
- An Area evolves or devolves to a new stage (re-generation of description)
- A zone state change significantly affects an existing Area (reactive update)

**Context bundle per generation call** (see `design/world-structure.md` for full model):
- `room-building.md` skill as system prompt
- Current zone: biome, Coherence level, faction footprint, ecology
- Adjacent Area/Room: full description (primary driver of continuity)
- 2–3 prior Areas in direction of travel: abbreviated
- Evolution stage of the new Area
- Boundary flag + adjacent zone state if near zone edge
- Exploration state: first visit vs. previously explored territory

**Caching**: Generated descriptions are stored on the Area/Room node and served from the database on subsequent visits. Re-generation only occurs on evolution stage change or explicit world events — not on every entry.

**Risk**: Inconsistency between generated prose and structural data (exits, fixtures). The generator must receive the exit list and fixture list as constraints, not generate them freely. Structure first, prose second.

### 3. Lore Oracle

**What**: A `lore <topic>` command lets players ask in-character questions about the world. An LLM answers based on a curated lore document.

**Example**: `lore Who are the Iron Mages?` → several paragraphs of in-world lore

**Implementation**:
- `docs/lore/` directory: structured lore documents (factions, history, geography)
- Relevant chunks selected by keyword match before being injected into the prompt
- Response is in-character, from the perspective of a world sage

**Guardrails**: The model is explicitly told not to invent locations, items, or characters that aren't in the lore docs. Hallucinations here are world-breaking.

### 4. Dynamic Quest Generation

**What**: An NPC can offer quests that are procedurally generated by an LLM but validated against live world data before presentation.

**Flow**:
1. Player talks to a quest-giver NPC
2. Game collects context: player level, zone, nearby mobs, items in player inventory
3. LLM generates a quest proposal (go kill X of Y mob, bring me Z item)
4. Game engine validates: does mob Y exist in a reachable zone? Does item Z exist?
5. If valid: quest offered. If invalid: retry or fall back to static quest.
6. Quest state tracked in memory (not persisted yet)

**Complexity**: This is the highest-complexity use case. Defer until core quest infrastructure exists.

### 5. Combat Narration

**What**: Instead of `You hit the Orc for 12 damage.`, generate varied combat prose. `Your blade finds a gap in the orc's guard, drawing a ragged wound across its shoulder (12).`

**Approach**: Template + LLM variation. Generate a small pool of narration strings per (attacker_type, attack_type, outcome) and cache them. Rotate through the pool so combat doesn't feel repetitive without calling the API on every hit.

## Technical Architecture

### API Layer

A thin async Rust client wrapping the chosen API (Claude / OpenAI-compatible). Responsibilities:
- Prompt construction
- Timeout enforcement (hard cutoff: 5s, return fallback)
- Rate limiting / backoff
- Response validation (length, content policy)

### Model Selection

| Use case | Latency need | Quality need | Recommendation |
|----------|-------------|-------------|----------------|
| NPC dialogue | Medium (1–3s ok) | High | Claude API (Haiku for cost, Sonnet for quality) |
| Room descriptions | Low (cached) | Medium | Local model (ollama/llama) or Haiku |
| Lore oracle | Medium | High | Sonnet |
| Quest generation | High (can be async) | High | Sonnet |
| Combat narration | Very low (cached pool) | Low | Local or Haiku |

### Context Injection Pattern

```rust
struct LlmContext {
    system_prompt: String,      // persona, rules, world facts
    lore_chunks:   Vec<String>, // relevant lore selected by keyword
    history:       Vec<(String, String)>, // (player, npc) turn pairs
    player_input:  String,
}
```

## Open Questions

- **Local vs. API**: Running ollama locally eliminates latency and cost concerns but caps quality. Use local for high-frequency low-stakes (descriptions, combat narration), API for high-stakes (dialogue, lore, quests)?
- **Streaming**: Claude/OpenAI support streaming. MUD protocol is line-oriented. We could stream word by word with a small buffer, or buffer the full response and send at once. Streaming feels more natural for dialogue.
- **Moderation**: Player input goes into prompts. Need input sanitization and a content filter on output before it hits the wire.
- **Async integration**: LLM calls must not block the game loop. They need to be spawned as tasks with results sent back via channel. The player sees `"(thinking...)"` or a typing indicator while the call is in flight.
- **Persistence of generated content**: Do we cache generated room descriptions to disk? Generated quest text? What's the staleness policy?
